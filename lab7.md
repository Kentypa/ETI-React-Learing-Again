## Які категорії помилок виявляє TypeScript на етапі компіляції

TypeScript дозволяє зловити типові помилки ще до запуску коду:
Виклик неіснуючих властивостей чи методів (user.adress замість user.address).
Невідповідність типів аргументів у функціях (add(2, "text")).
Неправильне використання об’єктів чи масивів (наприклад, доступ до індексу, який не існує).
Відсутність обов’язкових полів у об’єктах.
У JavaScript такі помилки проявляються лише під час виконання (Runtime), що може призвести до крашу додатку.

## Generics у TypeScript

Generics — це універсальні типи, які дозволяють писати гнучкий та повторно використовуваний код.
У випадку з кастомним хуком useFetch<T>:
Ми можемо вказати, який тип даних очікуємо від API (PostType, UserType, тощо).
Це вирішує проблему "невідомого типу" і дозволяє отримати автодоповнення та перевірку типів у компоненті.
Без Generics довелося б використовувати any, що знімає всі гарантії типізації.

## Різниця між any та unknown

any: повністю відключає перевірку типів. Значення можна використовувати як завгодно, і компілятор не попередить про помилки. Це порушує парадигму суворої типізації.
unknown: означає "невідомий тип", але вимагає явної перевірки перед використанням.
Наприклад, у блоці catch (error: unknown) ми мусимо перевірити, чи це Error, чи інший тип, перш ніж звертатися до його властивостей.
Використання unknown у catch захищає від неконтрольованого доступу до властивостей і робить код безпечнішим.

## Навіщо використовують успадкування в інтерфейсах компонентів

Приклад:

```tsx
ts;
interface InputProps extends React.InputHTMLAttributes<HTMLInputElement> {
  label: string;
}
```

Це дозволяє компоненту Input автоматично підтримувати всі стандартні HTML-атрибути (placeholder, disabled, onChange тощо).
Переваги:
Повна сумісність із нативними елементами.
Розширюваність — можна додати власні пропси (label, errorMessage).
Єдина дизайн-система — усі інпут-компоненти працюють однаково, але мають додаткові можливості.

## Чому useState<User | null>(null)

Дані користувача приходять асинхронно з API, тому початково їх немає.
Якщо ми оголосимо useState<User>(null), це буде помилка — null не є User.
Використання User | null дозволяє:
Безпечно перевіряти if (!user) return ... перед рендерингом.
Захиститися від ситуації, коли JSX намагається звернутися до властивостей user, а він ще не завантажений.
Це гарантує, що ми не отримаємо Runtime error "Cannot read property of null".
