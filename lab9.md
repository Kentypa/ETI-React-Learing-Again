# Різниця між src та dist (Процес збірки):

У src лежить вихідний код, зручний для розробника: TypeScript (який браузер не розуміє), JSX/TSX (React-розмітка), безліч окремих модулів і залежностей у node_modules.

У dist лежить згенерований (Bundle) код для браузера (Production). Між цими кроками Vite (через esbuild/Rollup) робить транскомпіляцію (TS -> JS, JSX -> JS), мініфікацію (видалення пробілів, коментарів), Tree-shaking (видалення мертвого коду) та розбиття на чанки (Code Splitting). Результат — легкі статичні файли .html, .css, .js.

# Філософія та переваги CI/CD порівняно з FTP:

FTP (Ручний): Повільно, є людський фактор (можна забути збілдити, забути прогнати тести), небезпечно (немає історії версій розгортання).

CI/CD (Автоматизований): Гарантує, що код відповідає стандартам якості (пройдено лінтер, тести) перед тим, як потрапити на прод. Деплой відбувається автоматично на основі тригерів (Push), що пришвидшує доставку цінності (Time-to-Market) і дозволяє команді фокусуватися на коді, а не на інфраструктурі.

# npm ci vs npm install (або yarn install --frozen-lockfile):

npm install може оновити версії пакетів, якщо вони вказані з "кареткою" (напр., ^1.2.3), що може зламати білд на сервері через невідповідність версій розробника і CI. npm ci (або yarn install --frozen-lockfile) жорстко встановлює точні версії з файлу package-lock.json (або yarn.lock), ігноруючи package.json. Це гарантує 100% відтворюваність середовища.

# Проблема 404 у SPA (React Router):

В SPA існує лише один фізичний HTML-файл — index.html. При кліку по посиланнях всередині додатку, React Router використовує window.history.pushState браузера, щоб змінити URL без перезавантаження сторінки, і динамічно малює нові компоненти. Але якщо користувач натискає F5 на [mysite.com/profile](https://mysite.com/profile), браузер робить реальний HTTP-запит до сервера (Vercel/GitHub Pages), просячи файл /profile/index.html. Сервер не знаходить такого файлу і видає 404. vercel.json каже серверу: "Віддай index.html для будь-якого шляху, а React Router сам розбереться".

# Чому конфіг .yml пишеться до злиття PR:

Це концепція "Shift-Left Security/Quality". Ми повинні блокувати поганий код (з помилками лінтера, впавшими тестами) ще на етапі Pull Request, щоб гілка main завжди залишалася стабільною ("Green"). Якщо ми налаштуємо CI після злиття, ми ризикуємо задеплоїти зламаний додаток у Production. Обов'язкові метрики: проходження всіх юніт-тестів, відсутність TypeScript-помилок (tsc --noEmit), дотримання кодстайлу (eslint).
